# Syl'slab

Syl'slab 是一套基于 Hermes Agent 的任务对话与上下文管理模式切换方案。

- **操作系统**: Windows 10
- **Agent**: Hermes Agent
- **大模型**: DeepSeek v4 Flash（单次输出上限 8192 tokens）
- **Shell**: git-bash (MSYS)

## 解决的问题

### 1. 单次输出超长

AI 模型的单次回复有输出上限（如 DeepSeek 为 8192 tokens），当一轮对话中要执行多个步骤、读取大文件、生成大量内容时，容易在中途被截断，导致任务不完整。

### 2. 任务跨上下文衔接

每轮对话的上下文窗口有限（1M tokens），当累积对话过长时，最早的消息会被裁剪。长任务需要在多个会话间无缝继续，而不是从头再来。

处理办法应该优于最近版本 WorkBuddy 的 context engineering 方案，因为无论对话还是 memory 都在做本地保存（虽然存储方式有待继续开发，但这是做本地化和永久记忆的一次尝试）。

## 三种模式

### 💻 开发者模式 (dev-mode)

**步骤定义**

步骤是按任务逻辑自动切分的。一个步骤包含完成一个最小功能单元所需的全部操作——可能包含多次工具调用（读文件、改文件、运行测试），但这些调用都属于同一个步骤。

步骤切分的依据：
- 按功能边界：写一个函数、创建一个文件、修复一个 bug 各算一步
- 按确认需求：如果某件事做完你需要看一眼结果，它就应该是一个独立步骤
- 不建议细到单行代码，也不建议粗到整个模块

**执行前**

每轮开头展示完整的任务全景，让你知道当前进度：
```
📋 <任务名>（共 M 步）
  已完成: ✅ 步骤A  ✅ 步骤B
  当前:   🔄 第 X/M 步：<当前步骤>
  待完成: ○ 步骤C  ○ 步骤D
```

**执行后**

展示当前步骤的执行结果和修改后的文件内容：
```
📄 修改后的文件内容：
  ┌─ path/to/file.py ────────────────────
  │ 完整文件内容展示在这里
  └──────────────────────────────────────
```

**确认后**

你确认后（输入"继续"），进入下一步。不确认就不动。

**输出保护**

与巡航者模式共享 `cruiser-calibrate.py` 的输出长度检测。进入模式时自动校准模型上限，每步脑内统计工具调用次数和输出大小。如果单步输出接近上限，会提醒你但依然等你确认，不会自动跳过。

**统计堆叠**

每次工具调用的输出大小、文字长度记录到 `usage-log.json`，同类操作的均值用于后续任务的估算。

**适合的场景**

- 写代码时每一步都想看到改了什么
- 调试时下一步取决于当前输出
- 高风险操作需要逐级确认
- 这是默认模式——新会话未选择时默认进入此模式

### 👁️ 巡航者模式 (cruiser-mode)

**阶段定义**

任务按逻辑拆成阶段（phase），每个阶段包含多个步骤。阶段内的步骤自动执行，不每步确认。阶段的边界由任务的自然功能划分决定——用户模块是一个阶段、支付模块是另一个阶段。

阶段切分的依据：
- 按功能域：用户系统、订单系统、报表系统各算一个阶段
- 按确认需求：每个阶段做完你应该看一次结果
- 一个阶段通常包含 3-8 个步骤

**执行中**

阶段内自动执行，每步结束后不暂停，直接进入下一步。每轮开头展示当前进度和输出估算：

```
📋 <任务名>（共 M 步）
  已完成: ✅ 步骤A  ✅ 步骤B
  当前:   🔄 步骤C
  当前阶段: 用户模块（4步中已做2步）
  输出估算: ~3500 / <soft_stop> ✅
→ 自动继续下一步骤
```

**输出保护**

实时估算本轮累计输出长度。每调用一个工具时脑内统计输出大小和文字长度。当估算值达到软停止阈值时，不再启动新步骤，收尾当前步骤后展示检查点。

软停止阈值来自 `cruiser-calibrate.py` 的自校准机制——进入模式时自动检测当前模型的上限（如 DeepSeek v4 Flash 为 8192 tokens），设为 85%（~6963 tokens）作为安全停止线。

如果实际发生截断，记录截断信息并自动下调阈值，多次校准后收敛到可靠值。

**阶段完成时**

阶段内所有步骤执行完毕，展示阶段总结：
```
📋 第一阶段完成 ✅
  阶段: 用户模块
  已完成: 注册接口 / 登录接口 / JWT中间件 / 权限校验（4步）
  改了: api/auth.py, middleware/jwt.py
  关键决定: RS256 签名
→ 继续到第二阶段？
  → "继续" / "换个方向" / "回滚到阶段1开始前"
```

**单步阶段**

如果某个阶段只有 1 个步骤，行为等同于开发者模式——做完即停，等你确认。两种模式的差异只在多步阶段中体现。

**输出上限时**

```
⚠️ 输出估算已达 ~6800 / 6963 tokens
已完成当前步骤，不再启动新步骤。
📋 阶段内检查点
  已完成: 注册接口 ✅  登录接口 ✅
  待完成: JWT中间件 ○  权限校验 ○
→ 开始新轮继续当前阶段？
  → "继续" / "回滚到阶段1开始前"
```

**统计堆叠**

与开发者模式相同：每次工具调用记录到 `usage-log.json`，同类操作取 5 次均值归类。数据用于未来任务的输出估算。

**适合的场景**

- 有多步骤的批量任务，想看到中间结果但不需每步确认
- 需要中间审查，但不想像开发者模式那样每步都停
- 比开发者模式快，但比长任务模式可控

### 🏃 长任务模式 (long-task-mode)

先出蓝图（列出阶段、步骤、预估桥接次数），你确认后开始全自动执行。每轮输出接近上限或阶段完成时，自动压缩对话上下文到 `history/` 目录，创建一个一次性 cron job，在 15 秒后触发新会话读取压缩文件继续执行。如此循环直到所有阶段完成。

执行过程中你完全不需要在场。完成后写入 `task.json.daily_log` 并通过 Windows 系统通知弹窗提醒你。

可以随时手动终止（设置 `cancel_requested` 标记 + 删除桥接 cron job）。

桥接次数的预估基于 `usage-log.json` 中同类任务的历史统计数据，没有数据时标注为未知，首次执行后自动校准。

## 三种模式对比

| | 开发者模式 | 巡航者模式 | 长任务模式 |
|---|---|---|---|
| 每步确认 | ✅ 是 | ❌ 阶段内自动 | ❌ 自动桥接 |
| 检查点 | 每步 | 阶段完成或快到上限 | 自动压缩+桥接 |
| 输出保护 | 到上限时提醒 | 到上限时暂停 | 到上限时自动桥接 |
| 你在场 | 需要 | 需要审阶段 | 不需要 |
| 多会话 | 不支持 | 手动压缩 | 自动桥接 |
| 最佳场景 | 交互式开发 | 批量任务审查 | 后台长期运行 |

## 快速开始

将 `skills/` 目录放入 Hermes Agent 可访问的路径下，每次新会话启动时通过模式选择器选取当前任务合适的模式。

三种模式共享同一套输出长度检测和统计堆叠机制。每次工具调用的输出大小、文字长度都会被累计记录到 `usage-log.json`，同类操作的均值用于未来任务的桥接次数预估——没有预设公式，完全基于实际经验。
