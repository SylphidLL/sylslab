# sylslab-task-modes

Syl'slab 项目下的任务对话与上下文管理模式切换 skill 包。

- **操作系统**: Windows 10
- **Agent**: Hermes Agent
- **大模型**: DeepSeek v4 Flash（单次输出上限 8192 tokens）
- **Shell**: git-bash (MSYS)

## 解决的问题

### 1. 单次输出超长

AI 模型的单次回复有输出上限（如 DeepSeek 为 8192 tokens），当一轮对话中要执行多个步骤、读取大文件、生成大量内容时，容易在中途被截断，导致任务不完整。

### 2. 任务跨上下文衔接

每轮对话的上下文窗口有限（1M tokens），当累积对话过长时，最早的消息会被裁剪。长任务需要在多个会话间无缝继续，而不是从头再来。

处理办法应该优于最近版本 WorkBuddy 的 context engineering 方案，因为无论对话还是 memory 都在做本地保存（虽然存储方式有待继续开发，但这是做本地化和永久记忆的一次尝试）。

## 三种模式

### 💻 开发者模式 (dev-mode)

**步骤定义**

步骤是按任务逻辑自动切分的。一个步骤包含完成一个最小功能单元所需的全部操作——可能包含多次工具调用（读文件、改文件、运行测试），但这些调用都属于同一个步骤。

步骤切分的依据：
- 按功能边界：写一个函数、创建一个文件、修复一个 bug 各算一步
- 按确认需求：如果某件事做完你需要看一眼结果，它就应该是一个独立步骤
- 不建议细到单行代码，也不建议粗到整个模块

**执行前**

每轮开头展示完整的任务全景，让你知道当前进度：
```
📋 <任务名>（共 M 步）
  已完成: ✅ 步骤A  ✅ 步骤B
  当前:   🔄 第 X/M 步：<当前步骤>
  待完成: ○ 步骤C  ○ 步骤D
```

**执行后**

展示当前步骤的执行结果和修改后的文件内容：
```
📄 修改后的文件内容：
  ┌─ path/to/file.py ────────────────────
  │ 完整文件内容展示在这里
  └──────────────────────────────────────
```

**确认后**

你确认后（输入"继续"），进入下一步。不确认就不动。

**输出保护**

与巡航者模式共享 `cruiser-calibrate.py` 的输出长度检测。进入模式时自动校准模型上限，每步脑内统计工具调用次数和输出大小。如果单步输出接近上限，会提醒你但依然等你确认，不会自动跳过。

**统计堆叠**

每次工具调用的输出大小、文字长度记录到 `usage-log.json`，同类操作的均值用于后续任务的估算。

**适合的场景**

- 写代码时每一步都想看到改了什么
- 调试时下一步取决于当前输出
- 高风险操作需要逐级确认
- 这是默认模式——新会话未选择时默认进入此模式

### 👁️ 巡航者模式 (cruiser-mode)

**阶段定义**

阶段是按单次回复输出上限切分的。一个阶段包含在当前回复的 soft_stop 阈值内能安全完成的一组步骤。

阶段内步骤自动执行，不每步确认。当累计输出接近 soft_stop（~6963 tokens）时，不再启动新步骤，收尾当前步骤后展示检查点。进入下一轮会话继续执行当前阶段的剩余步骤，直到阶段完成。

阶段切分的依据：
- 按输出容量：每轮回复的 soft_stop 以内能做完的事算一个阶段
- 阶段边界 = soft_stop 触发点，不是功能模块边界
- 一个功能模块可能横跨多个阶段，一个阶段也可能包含多个小功能

**执行中**

阶段内自动执行，每步结束后不暂停，直接进入下一步。每轮开头展示当前进度和输出估算：

```
📋 <任务名>（共 M 步）
  已完成: ✅ 步骤A  ✅ 步骤B
  当前:   🔄 步骤C
  当前阶段: 用户模块（4步中已做2步）
  输出估算: ~3500 / <soft_stop> ✅
→ 自动继续下一步骤
```

**输出保护**

实时估算本轮累计输出长度。每调用一个工具时脑内统计输出大小和文字长度。当估算值达到软停止阈值时，不再启动新步骤，收尾当前步骤后展示检查点。

软停止阈值来自 `cruiser-calibrate.py` 的自校准机制——进入模式时自动检测当前模型的上限（如 DeepSeek v4 Flash 为 8192 tokens），设为 85%（~6963 tokens）作为安全停止线。

如果实际发生截断，记录截断信息并自动下调阈值，多次校准后收敛到可靠值。

**阶段完成时**

阶段内所有步骤执行完毕，展示阶段总结：
```
📋 第一阶段完成 ✅
  阶段: 用户模块
  已完成: 注册接口 / 登录接口 / JWT中间件 / 权限校验（4步）
  改了: api/auth.py, middleware/jwt.py
  关键决定: RS256 签名
→ 继续到第二阶段？
  → "继续" / "换个方向" / "回滚到阶段1开始前"
```

**单步阶段**

如果某个阶段只有 1 个步骤，行为等同于开发者模式——做完即停，等你确认。两种模式的差异只在多步阶段中体现。

**输出上限时**

```
⚠️ 输出估算已达 ~6800 / 6963 tokens
已完成当前步骤，不再启动新步骤。
📋 阶段内检查点
  已完成: 注册接口 ✅  登录接口 ✅
  待完成: JWT中间件 ○  权限校验 ○
→ 开始新轮继续当前阶段？
  → "继续" / "回滚到阶段1开始前"
```

**统计堆叠**

与开发者模式相同：每次工具调用记录到 `usage-log.json`，同类操作取 5 次均值归类。数据用于未来任务的输出估算。

**适合的场景**

- 有多步骤的批量任务，想看到中间结果但不需每步确认
- 需要中间审查，但不想像开发者模式那样每步都停
- 比开发者模式快，但比长任务模式可控

### 🏃 长任务模式 (long-task-mode)

**蓝图定义**

执行前先出蓝图，列出所有阶段和步骤，以及预估的桥接次数。你确认后才开始执行。桥接次数的预估基于 `usage-log.json` 中同类任务的历史统计数据——没有预设公式，完全基于实际经验。没有数据时标注为未知，首次执行后自动校准。

蓝图格式：
```
🏃 长任务计划：<任务名>

阶段1：<阶段名>
  1. <步骤描述>
  2. <步骤描述>

阶段2：<阶段名>
  3. <步骤描述>
  ...

桥接预估: ~N 次（基于 N 条历史记录）| 未知（无历史数据）
```

**执行中**

每轮自动执行当前阶段内的步骤。如果是首轮（新任务），直接从步骤 1 开始。如果是桥接恢复的会话，先读取 `task.json.latest_history` 指向的压缩文件恢复上下文，然后继续。

**桥接触发**

以下任一条件触发桥接：
- **当前阶段完成**：阶段内所有步骤执行完毕
- **输出接近上限**：累计输出达到 soft_stop

**桥接流程**

触发桥接时自动执行：
1. 压缩当前进度到 `history/context-<任务名>-YYYY-MM-DD-N.json`（带 bridge_id / total_bridges 统计）
2. 更新 `task.json` 的 `latest_history` 指针
3. 创建一个一次性 cron job（15 秒后触发），内容为"读取最新的 history 文件继续执行长任务"
4. 当前轮结束

15 秒后 cron job 触发 → 新会话启动 → 加载 `sylphid-skill-bridge` → 读取 `latest_history` → 恢复上下文 → 继续执行 → 回到执行中循环。

**完成通知**

所有阶段执行完毕后：
1. 写入 `task.json.daily_log`（持久记录）
2. 发送 Windows 系统通知弹窗（`NotifyIcon` balloon tip）
3. 不在当前对话中发送消息，避免与已有对话冲突

**中途终止**

任何时候可以说"终止任务"或"取消长任务"：
1. 设置 `task.json.cancel_requested = true`
2. 删除该任务的所有桥接 cron job
3. 后续 cron job 启动时检测到 `cancel_requested` 标记，自动清理退出

**执行信息查询**

随时可以问我"长任务跑完了吗"或"看看长任务进度"，我会扫描 `daily_log` 和 `history/` 目录报告最新状态。

**适合的场景**

- 需要在多个会话间无缝继续的超长任务
- 设定后不需要你在场的批处理
- 定期自动执行的后台任务

## 三种模式对比

| | 开发者模式 | 巡航者模式 | 长任务模式 |
|---|---|---|---|
| **步骤确认** | 每步都等确认 | 阶段内自动，阶段边界停 | 全自动，不等待 |
| **切分依据** | 功能边界 | 单次输出上限（soft_stop） | 功能模块+输出上限 |
| **输出保护** | 到上限时提醒，仍等你确认 | 到上限时暂停，展示检查点 | 到上限时自动压缩+桥接 |
| **跨会话** | 不支持 | 手动压缩后可继续 | 自动桥接，无感衔接 |
| **你在场** | 需要全程 | 需要审阶段 | 不需要 |
| **通知方式** | 对话内展示 | 对话内检查点 | daily_log + 系统弹窗 |
| **最佳场景** | 交互式开发、逐行审查 | 批量任务、需中间把关 | 超长任务、后台运行 |

## 快速开始

将 `skills/` 目录放入 Hermes Agent 可访问的路径下，每次新会话启动时通过模式选择器选取当前任务合适的模式。

三种模式共享同一套输出长度检测和统计堆叠机制。每次工具调用的输出大小、文字长度都会被累计记录到 `usage-log.json`，同类操作的均值用于未来任务的桥接次数预估——没有预设公式，完全基于实际经验。
